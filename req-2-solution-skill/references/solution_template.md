## 二、技术选型对比

> 说明：以下对各层关键技术进行备选方案对比，给出推荐结论。
> 推荐原则：优先选成熟开源 + 社区活跃 + 迁移成本低；高风险自研留到差异化核心竞争力环节。

### 2.1 应用框架层

| 方案 | 适用场景 | 优点 | 缺点 | 推荐 |
|------|----------|------|------|------|
| Spring Boot / Spring Cloud | Java 技术栈，中大型系统 | 生态完善，插件丰富，企业级 | 较重，学习曲线 | ⭐ 推荐 |
| FastAPI / Django | Python 技术栈，AI 密集型 | 轻量，异步，性能好，AI 集成方便 | 高并发不如 Java | ⭐ 推荐（AI 推理层）|
| NestJS | TypeScript 技术栈 | 全栈统一，装饰器风格，适合微服务 | 社区比 Java 小 | 可选 |

### 2.2 数据存储层

| 方案 | 适用场景 | 优点 | 缺点 | 推荐 |
|------|----------|------|------|------|
| PostgreSQL + TimescaleDB | 时序 + 关系混合数据 | SQL 生态丰富，时序扩展好 | 需要两个引擎协同 | ⭐ 推荐（综合型）|
| MySQL + InfluxDB | 轻量时序场景 | MySQL 运维成熟，InfluxDB 轻量 | 两套系统，增加运维复杂度 | 备选 |
| MongoDB | 非结构化数据多 | 灵活，Schema-less | 聚合查询弱，不适合强事务 | 特定场景可选 |

### 2.3 消息队列层

| 方案 | 适用场景 | 优点 | 缺点 | 推荐 |
|------|----------|------|------|------|
| Apache Kafka | 高吞吐，日志流，事件溯源 | 生态完善，吞吐量高 | 运维复杂度较高 | ⭐ 推荐 |
| RabbitMQ | 中小吞吐量，路由灵活 | 上手简单，功能丰富 | 吞吐瓶颈明显 | 备选（轻量场景）|
| RocketMQ | 电商交易场景 | 阿里背书，事务消息强 | 社区比 Kafka 小 | 特定场景可选 |

### 2.4 AI / 大模型层

| 方案 | 适用场景 | 优点 | 缺点 | 推荐 |
|------|----------|------|------|------|
| 云端大模型 API（DeepSeek / GPT-4 / 智谱）| 快速上线，无需 GPU | 零运维，模型能力强 | 数据合规风险，有 token 成本 | ⭐ 推荐（通用场景）|
| 开源模型私有化部署（Qwen / LLaMA / ChatGLM）| 数据合规要求高 | 数据不出网，可微调 | 运维成本高，硬件要求高 | ⭐ 推荐（合规敏感场景）|
| LangChain / LangGraph（智能体编排）| 多智能体协同 | 生态成熟，支持多种 LLM | 有一定学习成本 | ⭐ 推荐 |
| RAG（检索增强生成）| 知识库问答 | 结合私有知识，回答准确 | 依赖检索质量 | ⭐ 推荐 |

### 2.5 知识图谱层

| 方案 | 适用场景 | 优点 | 缺点 | 推荐 |
|------|----------|------|------|------|
| Neo4j 企业版 | 中大规模知识图谱 | 功能完善，可视化好 | 商业版收费，大规模成本高 | ⭐ 推荐 |
| NebulaGraph | 超大规模图数据 | 性能好，开源，可私有化 | 生态比 Neo4j 小 | ⭐ 推荐（大规模场景）|
| 自建（PostgreSQL + 图扩展）| 轻量图查询 | 零额外成本 | 功能有限 | 轻量场景可选 |

### 2.6 基础设施层

| 方案 | 适用场景 | 优点 | 缺点 | 推荐 |
|------|----------|------|------|------|
| Kubernetes + Docker | 生产级微服务 | 弹性扩缩，生态完善 | 运维门槛高 | ⭐ 推荐 |
| Docker Compose | 小规模验证 | 简单，快速上手 | 无法弹性扩缩 | 轻量验证可选 |
| 阿里云 ACK / 腾讯云 TKE | 不想自建 K8s | 托管 K8s，省运维 | 成本较高，供应商锁定 | 云优先团队可选 |

### 2.7 前端展示层

| 方案 | 适用场景 | 优点 | 缺点 | 推荐 |
|------|----------|------|------|------|
| React + Ant Design Pro | 企业管理后台 | 组件丰富，文档完善 | 较重 | ⭐ 推荐 |
| Vue + Element Plus | 快速开发 | 上手简单，中文文档好 | 生态比 React 小 | 可选 |
| Flutter / Tauri（桌面端）| 需要桌面客户端 | 原生体验，跨平台 | 投入成本高 | 特定场景可选 |

> - L1 = 用户可感知的一个独立业务模块

## 实现难点等级说明

| 等级 | 说明 |
|------|------|
| 低 | 有成熟开源方案，实现路径清晰，团队有相关经验 |
| 中 | 需要一定定制开发，有少量不确定性，需要调研验证 |
| 高 | 无成熟方案，需要创新或深度定制，跨领域技术整合，风险较高 |

## 功能拆分原则

1. **L1 功能** = 用户可感知的一个独立业务模块（如"用户管理""订单管理"）
2. **L2 功能** = L1 下的具体操作项（如"用户管理 > 新增用户""用户管理 > 批量导入"）
3. 每个 L2 功能须包含：功能名称、简单描述、实现难点、预估工时
4. 如果需求较简单，可只拆分到 L1；但要有完整的描述和工时

### 必选 vs 可选判断原则

| 类型 | 判断标准 | 典型设备 |
|------|---------|---------|
| ⭐ 必选 | 系统正常运行和数据安全的核心依赖 | 应用服务器、数据库服务器、核心交换机、防火墙、备份存储、UPS |
| ⭐ 可选 | 业务非核心依赖，或有低成本替代方案时可选 | GPU 服务器（云端 API 可替代）、硬件负载均衡（软件可替代）、工业交换机（复用现有网络）、VPN 网关（复用现有 VPN）|
