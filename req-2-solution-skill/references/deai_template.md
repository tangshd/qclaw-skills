# 去 AI 化（Humanize）标准

每次生成解决方案终稿后，必须经过去 AI 化处理，消除 AI 生成痕迹，提升文档专业度和可信度。

---

## 去 AI 化原则

**核心目标：** 让文档读起来像**经验丰富的行业专家亲笔撰写**，而非 AI 批量生成。

- **语气**：坚定、果断、有主见，不用"可能"、"也许"、"大概"等模糊词
- **结构**：层次分明，逻辑递进，符合商业文档阅读习惯
- **表达**：主动语态，结论前置，避免"首先...然后...最后"式机械罗列
- **数据**：具体数字优先于"若干"、"多个"等模糊量词

## AI 口吻识别清单（必须消除）

### 一、模糊度词 — 全部替换

| ❌ AI 常用表达 | ✅ 专业替换 |
|---------------|-----------|
| 可能 / 也许 / 或许 / 大概 | （直接删掉，或改为具体说明） |
| 一定程度上 / 在某种程度上 | （删掉或改为具体描述） |
| 一些 / 若干 / 部分 / 多个 | 替换为具体数量或范围 |
| 通常 / 一般来说 / 常规情况下 | 删掉，或改为具体场景描述 |
| 相对 / 比较 / 较为 / 较为 | 直接描述实际程度 |
| 大约 / 左右 / 上下 | 替换为精确数字或合理范围 |
| 一般而言 / 通常来说 / 总体而言 | 删掉 |
| 可以看出 / 据分析 / 通过分析可知 | 改为客观陈述（"方案采用 X 技术"） |
| 值得注意的是 / 需要注意的是 | 改为"本方案"、"本项目"开头的陈述 |
| 综上所述 / 总而言之 | 改为"建议"或删掉，直接进入结论 |

### 二、AI 结构词 — 优先消除

| ❌ AI 常用表达 | ✅ 专业替换 |
|---------------|-----------|
| 首先、其次、再次、最后 | 改为逻辑递进词或直接分点 |
| 第一、第二、第三（滥用时）| 保留数字序号，但不要滥用 |
| 具体来说 / 详细而言 | 直接列出内容 |
| 换句话说 / 也就是说 | 改为直接说明 |
| 比如说 / 例如 | 有具体数据时直接用数据；无数据时删掉 |

### 三、AI 人称词 — 消除第一人称

| ❌ AI 常用表达 | ✅ 专业替换 |
|---------------|-----------|
| 我认为 / 我觉得 / 我们建议 | 改为"本方案"、"建议"、"推荐" |
| 我的看法是 | 删掉 |
| 我们团队经过分析 | 改为"方案采用" |
| 结合上述分析 / 综合以上 | 改为"基于以上分析，本方案" |
| 您可以 / 你可以 | 改为"建议"或"推荐" |

### 四、AI 填充词 — 删除冗余

| ❌ AI 常用表达 | ✅ 专业替换 |
|---------------|-----------|
| 当然 / 无疑 / 毋庸置疑 | 删掉 |
| 实际上 / 事实上 | 改为直接陈述 |
| 非常 / 极其 / 十分（滥用时）| 删掉或改为具体描述 |
| 必不可少 / 不可或缺 | 改为"必要"或"核心" |
| 不言而喻 / 不言自明 | 删掉 |
| 简而言之 / 长话短说 | 直接说内容 |
| 值得注意的是 / 重点强调的是 | 改为直接陈述 |

### 五、AI 过度保证词 — 降低绝对性

| ❌ AI 常用表达 | ✅ 专业替换 |
|---------------|-----------|
| 确保 / 保证 / 一定能 | 改为"确保（采取 XX 措施后）" |
| 绝对不会 / 绝不可能 | 改为"风险可控" |
| 完美 / 万无一失 / 无懈可击 | 改为"稳定可靠" |
| 彻底解决 / 完全消除 | 改为"有效降低" |

### 六、AI 风险描述词 — 适度调整

| ❌ AI 常用表达 | ✅ 专业替换 |
|---------------|-----------|
| 存在一定风险 / 存在一定难度 | 存在[具体维度]风险，表现为[具体描述]，应对策略为 |
| 可能面临的挑战 | 实施挑战包括 |
| 需要关注的重点 | 本方案关键点包括 |

## 各章节去 AI 化要点

### 一、整体解决方案

- ❌ "我认为本方案适合贵公司的需求"
- ✅ "本方案针对贵公司[具体业务场景]，采用[技术路线]，可实现[量化目标]"

- ❌ "从架构图中可以看出..."
- ✅ 直接描述架构，不加"可以看出"

### 二、技术选型对比

- ❌ "根据我们的分析，Spring Boot 是最佳选择"
- ✅ "推荐 Spring Boot，理由如下：①生态成熟... ②社区活跃..."

- ❌ "总的来说，PostgreSQL 在大多数情况下表现良好"
- ✅ "PostgreSQL 适用于[场景]，优势在于[具体说明]"

### 三、功能拆分表

- 工时估算行中，去掉"大约"、"左右"
- 难点描述不用"存在一定难度"，改为"需要[具体工作内容]"

### 四、风险与约束

- ❌ "可能存在技术风险"
- ✅ "技术风险：方案依赖[具体技术]，若[具体条件]，可能导致[后果]；应对：[具体策略]"

### 五、硬件配置

- ❌ "配置大约需要 50 万左右"
- ✅ "配置方案估算总投入约 45～55 万元（含税）"

## 去 AI 化检查流程

1. **通读全文** — 识别所有 AI 口吻关键词
2. **逐项替换** — 按本章清单逐项替换
3. **语气检查** — 读出声来，确保语气坚定、不含糊
4. **数据核验** — 确认所有数字具体、无模糊量词
5. **终稿输出** — 去 AI 化后的文档即为终稿
