# 解决方案审核标准

每次生成解决方案后，必须按本标准进行自审，并输出审核结论。

---

## 审核流程

1. **逐维度评分** — 对方案各部分按 1～5 分打分
2. **识别问题** — 发现问题记录问题描述、严重程度、涉及章节
3. **修改完善** — 针对问题对方案进行修改
4. **输出终稿** — 将审核结论（含评分、问题清单、修改记录）附在文档末尾

---

## 审核维度与评分标准

| 审核维度 | 权重 | 评分说明（1=差，5=优秀） |
|----------|------|------------------------|
| 需求匹配度 | 25% | 方案是否精准回应了客户的核心诉求，有无遗漏或偏离 |
| 架构合理性 | 20% | 技术选型是否合理，架构是否能支撑业务规模，扩展性如何 |
| 功能完整性 | 20% | 功能拆分是否覆盖全部需求，L1/L2 层级是否清晰无遗漏 |
| 逻辑自洽性 | 15% | 各章节之间描述是否一致，前后矛盾是否存在 |
| 工时合理性 | 10% | 工时估算是否与功能复杂度匹配，有无严重高估或低估 |
| 可实施性 | 10% | 方案是否有实施路径，里程碑是否合理，风险是否可控 |

**加权总分 = Σ(各维度得分 × 权重)**

---

## 问题严重程度定义

| 等级 | 定义 | 处理方式 |
|------|------|----------|
| 🔴 高（阻断）| 会导致项目失败或客户严重不满的问题 | 必须修改后才能交付 |
| 🟡 中（建议）| 影响方案质量但不影响项目可行性的问题 | 建议修改，可与客户协商 |
| 🟢 低（优化）| 优化类建议，不影响当前方案 | 视情况决定是否修改 |

---

## 自审问题清单（逐项检查）

### 一、整体解决方案
- [ ] 方案定位是否与客户需求的核心痛点匹配？
- [ ] 技术路线选择是否有明确依据（业务适配、技术成熟度、团队能力）？
- [ ] 多方案对比（A/B 方案）时，推荐方案的优势是否充分说明？
- [ ] 模块划分是否清晰，模块边界是否合理？

### 二、技术选型对比
- [ ] 每层技术选型是否有 2～3 个备选方案？
- [ ] 各方案优缺点是否客观、全面、无明显遗漏？
- [ ] 推荐结论是否有依据（不能只凭个人偏好）？
- [ ] 是否有数据合规、运维能力、迁移成本等非功能维度的考量？

### 三、功能拆分
- [ ] 所有需求点是否都有对应的功能拆分？
- [ ] L1 / L2 层级是否清晰、无越级或遗漏？
- [ ] 实现难点评估是否与实际技术风险匹配？
- [ ] 工时估算是否与难度等级相符？
- [ ] 是否存在"需求有提及但功能拆分中未覆盖"的内容？

### 四、非功能性需求
- [ ] 性能要求是否有量化指标（响应时间、并发数、数据量）？
- [ ] 安全要求是否覆盖了身份认证、权限控制、数据加密、传输安全？
- [ ] 兼容性要求是否覆盖了目标用户的使用环境（浏览器、系统、设备）？

### 五、风险与约束
- [ ] 风险识别是否覆盖了技术、进度、资源、合规等维度？
- [ ] 高风险项是否有具体的应对策略而非泛泛而谈？
- [ ] 是否识别了客户侧的约束条件（预算、时间、技术储备）？

### 六、总体工时估算
- [ ] 工时合计是否与功能拆分表的单项工时之和基本一致（允许±10%误差）？
- [ ] 各阶段工期分配是否合理（设计和开发占主要）？
- [ ] 是否留有适当的缓冲时间（建议设计阶段 +15%，开发阶段 +10%）？

### 七、实施里程碑
- [ ] 里程碑设置是否与客户期望的交付节奏匹配？
- [ ] 每个里程碑的交付物是否具体、可验证？
- [ ] 里程碑之间的依赖关系是否合理？

### 八、硬件配置
- [ ] 配置方案是否与系统规模（用户量、数据量、并发）匹配？
- [ ] 必选 vs 可选区分是否合理？
- [ ] 价格估算是否在合理区间（明显偏高或偏低都需要说明原因）？
- [ ] 分阶段采购建议是否与实施里程碑对齐？

## 交付判断标准

| 总分区间 | 交付状态 | 说明 |
|----------|----------|------|
| 4.0 ~ 5.0 | ✅ 可直接交付 | 方案质量优秀，轻微优化项不影响交付 |
| 3.0 ~ 3.9 | ⚠️ 建议修改后交付 | 存在中等或轻微问题，建议按审核意见优化 |
| < 3.0 | ❌ 需修改后再审 | 存在阻断性问题，必须修改并重新审核后方可交付 |
