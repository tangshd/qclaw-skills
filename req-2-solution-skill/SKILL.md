# req-2-solution-skill

根据客户需求文档，生成完整的软件系统解决方案文档。

## 输入处理

1. **文字粘贴**：直接读取，提取其中的业务需求、功能描述、约束条件
2. **上传文件**（PDF/Word/TXT）：读取文件内容，提取需求相关信息

> 若输入信息不足或存在歧义，先向用户提问确认，不要自行假设。

## 输出内容（共 8 项基础 + 按需扩展）

**基础内容（必出）：**
1. **整体解决方案** — 方案定位、技术架构建议、模块划分
2. **技术选型对比** — 各层关键技术备选方案对比（含推荐结论）
3. **功能拆分表** — 编号、L1/L2层级、功能名称、简单描述、实现难点（低/中/高）、预估工时
4. **非功能性需求** — 性能、安全、兼容性、可维护性
5. **风险与约束** — 风险项、等级、说明、应对策略
6. **总体工时估算** — 分阶段列出
7. **实施里程碑建议** — 关键里程碑 + 时间节点 + 交付内容
8. **硬件配置推荐** — 按规模分档，含必选/可选、规格建议、估算价格

**按需扩展内容（见下方触发关键词）：**
- A. 商务报价单（触发词：`报价` `预算` `商务` `多少钱` `成本` `招标` `投标` `人天`）
- B. 竞品差异化分析（触发词：`竞品` `对比` `差异化` `其他厂商` `vs` `优势` `选型对比`）
- C. 合规映射（触发词：`等保` `GDPR` `合规` `数据安全` `医疗认证` `金融监管` `政务` `信创` `国产化`）
- D. Q&A 预判（触发词：`QA` `答辩` `质疑` `评审` `FAQ` `常见问题` `汇报准备`）
- E. 执行摘要 + PPT（触发词：`PPT` `演示` `汇报` `摘要` `一页纸` `executive summary` `高管`）

## 输出格式

- 文件格式：Markdown（`.md`）
- 交付方式：生成文件保存至工作目录
- 表格使用 Markdown 表格语法

## 工作流程

**第一步：内容生成（1-7）**

1. **解析需求** → 提取业务背景、核心功能、性能要求、约束条件、目标规模
2. **方案设计** → 确定方案定位和技术方向（可提供 1～2 个方案对比）
3. **技术选型对比** → 针对每层关键技术提供 2～3 个备选方案，说明优缺点，给出推荐结论
4. **功能拆分** → 按 L1（一级模块）/ L2（二级功能）两层结构梳理
5. **工时估算** → 参考 `references/solution_template.md` 中的估算标准
6. **硬件配置** → 根据系统规模和用户数推荐配置，分轻量/标配/高配三档
7. **组装输出** → 按 `references/solution_template.md` 中的模板格式生成完整 Markdown 文档

**第二步：检测扩展项（基于关键词自动触发）**

> 在生成完基础文档后，检查用户输入和需求背景，识别需要生成的扩展内容：
> - 有「报价」类关键词 → 生成报价单（`references/quotation_template.md`）
> - 有「竞品」类关键词 → 生成竞品差异化分析（`references/competitive_template.md`）
> - 有「合规」类关键词 → 生成合规映射（`references/compliance_template.md`）
> - 有「Q&A」类关键词 → 生成 Q&A 预判（`references/qa_template.md`）
> - 有「PPT/汇报」类关键词 → 生成执行摘要 + PPT（`references/exec_summary_template.md`）

**第三步：品质管控（8-11）**

8. **痛点映射（必选）** → 按照 `references/painpoint_mapping.md` 中的标准，将每个功能模块与客户痛点逐一对应，标注覆盖状态，未覆盖项给出补充建议
9. **方案自审** → 按照 `references/audit_template.md` 中的审核标准，对生成方案进行逐项自审，给出评分与修改建议
10. **去 AI 化处理** → 按照 `references/deai_template.md` 中的标准，识别并消除全文中的 AI 口吻词、模糊词、第一人称、AI 填充词，替换为专业、坚定、有数据支撑的表达
11. **生成架构图** → 调用 `drawio-diagram` 技能，根据方案中的模块划分、技术选型，生成配套 `.drawio.svg` 架构图文件，附加到方案目录中

**第四步：输出终稿**

- 审核分 ≥ 3.0 + 去 AI 化完成 → 终稿可交付
- 审核分 < 3.0 → 退回修改，重复品质管控流程
- 所有扩展内容（如已触发）一并附在终稿对应章节后

## 审核流程（必须执行）

每次生成解决方案后，必须经过审核阶段方可交付：

1. **加载审核标准** → 读取 `references/audit_template.md`
2. **逐项评分** → 按审核维度对方案各部分打分（1-5 分）
3. **识别问题** → 列出发现的具体问题，标注严重程度
4. **修改完善** → 根据问题对方案进行针对性修改
5. **去 AI 化处理** → 读取 `references/deai_template.md`，对全文进行去 AI 化，消除口吻词、模糊词、冗余填充
6. **输出终稿** → 将去 AI 化后的终稿方案（含审核结论 + 去 AI 化记录 + 痛点映射）交付给用户

## 品质管控标准速查

| 品质阶段 | 依据文件 | 阻断条件 |
|---------|---------|---------|
| 痛点映射 | `painpoint_mapping.md` | ❌ 未覆盖痛点超过 2 项且无补充建议 → 阻断 |
| 方案自审 | `audit_template.md` | 审核总分 < 3.0 → 阻断，需修改后再审 |
| 去 AI 化 | `deai_template.md` | AI 口吻词 > 5 处未处理 → 阻断 |
| 架构图 | `drawio-diagram` 技能 | 节点数 < 3 → 可跳过，但建议生成 |
